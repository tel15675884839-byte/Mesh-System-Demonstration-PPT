# Wireless Mesh Fire Alarm System — HTML Presentation Blueprint

## Goal
Build a **full-screen slide-style HTML presentation** that feels like a product demo, not a copied PPT.

This blueprint is designed so you can **start simple and expand later**.

---

## Overall Structure
Recommended format: **single-page presentation app** with full-screen sections.

Suggested page flow:
1. Opening
2. Feature Overview
3. Extended Distance
4. Mesh Communication
5. Rapid Response + High Reliability
6. 10-Year Battery + Easy Installation
7. System Capacity
8. Coverage + Wall Penetration
9. Closing / Contact / Q&A

---

## Core Design Principle
Each page should contain only these 4 layers:

### 1. Main message
What is the one thing the audience should remember from this page?

### 2. Visual focus
What is the first thing they see?

### 3. Interaction
What can be clicked, switched, or animated to make the idea easier to understand?

### 4. Spoken explanation
What do you say while this page is on screen?

## Page 4 — Mesh Communication

### Purpose
Show the core value of real Mesh technology.

### On-screen content
- Title: **Mesh Communication**
- Key line: **Multiple communication paths**
- Value line: **More stable in complex buildings**
- Visual:
  - network nodes
  - active path highlighted
  - blocked path turns gray
  - alternative path lights up automatically

### Visual focus
A dynamic network topology.

### Interaction
Recommended controls:
- `Normal Mode`
- `Path Blocked`
- `Auto Recovery`

Expected behavior:
- In `Normal Mode`, signal follows default route
- In `Path Blocked`, one route is disabled
- In `Auto Recovery`, signal switches to another route automatically

Optional labels:
- Leader
- Node
- Panel
- Signal Path
- Backup Path

### Spoken explanation
Second, Mesh communication.

This is one of the biggest advantages of our system.

Unlike traditional wireless systems that may depend on one communication path, Mesh technology allows signals to travel through multiple paths.

If one path is blocked or weak, the signal can go through another path.

In simple words, the devices support each other in the network.

That is why the system is more reliable in real projects.

For customers, this means better communication stability, especially in complex buildings or difficult environments.

This is the real value of genuine Mesh technology.

## Page 7 — System Capacity

### Purpose
Show that the system has strong overall capacity and explain where the numbers come from.

### On-screen content
Top summary cards:
- **Up to 2,000 wireless devices**
- **Up to 128 nodes**
- **Up to 32 devices per leader/node**

Below that, show the calculation logic.

### Recommended layout
Three blocks:

#### Block 1 — Network Level
- 16 loops
- 125 devices per loop
- 16 × 125 = 2,000 devices

#### Block 2 — Loop Expansion Card
- 4 loops per card
- 32 nodes per loop
- 4 × 32 = 128 nodes

#### Block 3 — Device Level
- 1 wireless leader/node
- up to 32 devices

### Visual focus
Animated number build-up.

### Interaction
- Button: `Show Calculation`
- Expand animation reveals formulas step by step
- Optional tabs:
  - `Network Level`
  - `Loop Card`
  - `Device Level`

### Spoken explanation
Now let’s look at the system capacity.

These figures are based on the system structure, so let me explain how they are calculated.

First, at the network level, the system supports up to 16 wireless loops.

And each loop can support up to 125 devices.

So 16 multiplied by 125 gives a total of 2,000 wireless devices.

Second, for the loop expansion card, one card supports up to 4 loops.

And each loop can support up to 32 nodes.

So 4 multiplied by 32 gives a total of 128 nodes.

Finally, at the device level, each wireless leader or node can support up to 32 devices.

So these figures show that the system has a strong overall capacity and is suitable for large-scale projects.

### Expand later
- Add simplified network diagram
- Add tooltip definitions for loop / node / leader
- Add customer scenario examples by project size

# What Should Be On Screen vs What Should Be Spoken

## On screen
Keep it short:
- titles
- key numbers
- one-line value statements
- labels for diagrams

## Spoken
You can use your full explanation.

Rule:
**The screen should help the audience understand faster, not force them to read long paragraphs.**
